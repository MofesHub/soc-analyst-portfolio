## 1. Alert Summary

A medium severity alert fired on `WIN10-SOC-LAB` for Event ID 4702 — a scheduled task modification. Scheduled task creation and modification are commonly abused by attackers for persistence (MITRE ATT&CK T1053.005), making this a high-value event to monitor. Investigation of the task name, modifying account, timing, and associated process confirmed the activity was legitimate — a Windows system process performing a routine task update. Alert closed as a false positive.

## 3. Investigation Steps & Findings

### Step 1 — Splunk Query: Task Name & Modifier

**Finding:** The task name falls under the `\Microsoft\Windows\` namespace — Microsoft's reserved path for built-in Windows scheduled tasks. Modifications to tasks in this namespace by the `SYSTEM` account via `svchost.exe` are expected system behaviour, particularly during Windows Update cycles.

**Red flags present:** None. Task name is recognised, account is expected, process is legitimate.

### Step 2 — Process Tree Check

**Finding:** The modifying process was `svchost.exe` running under the `SYSTEM` account — the standard host process for Windows services. No unusual parent process, no command-line arguments, no child processes spawned as a result of the modification.

### Step 3 — Timing Analysis

**Finding:** The modification occurred during normal system uptime with no correlation to any other suspicious events in the same window. No failed logons, no new user accounts, no network anomalies within ±15 minutes.

### Step 4 — Baseline Comparison

**Finding:** Event ID 4702 fires routinely on Windows systems during update cycles. This event in isolation, with a known task name, SYSTEM account, and svchost.exe process, matches the established baseline for legitimate Windows maintenance activity.

---

## 4. Why This Is NOT Malicious

Attackers use scheduled tasks for persistence — but their tasks have a distinct fingerprint:

| Indicator | Malicious Pattern | This Case |
|---|---|---|
| Task Name | Random strings, misspelled system names, unusual paths | Legitimate Microsoft namespace |
| Modifying Account | Low-privilege user, service account, new account | SYSTEM — expected |
| Process | `powershell.exe`, `cmd.exe`, `wscript.exe` | `svchost.exe` — expected |
| Timing | Off-hours, outside maintenance windows | Normal uptime |
| Correlated Events | Other suspicious activity nearby | None |

None of the malicious indicators are present. This is a clean false positive.

---

## 5. Triage Decision & Reasoning

**Decision: FALSE POSITIVE — Close alert**

The 4702 event was generated by a legitimate Windows system process performing a routine scheduled task update. There is no evidence of attacker activity, persistence mechanism, or policy violation. Closing without escalation.

**Closing Note:**
> Event ID 4702 on `WIN10-SOC-LAB` confirmed false positive. Task `\Microsoft\Windows\UpdateOrchestrator\Reboot` modified by SYSTEM via svchost.exe during normal uptime. No correlated suspicious activity. No action required. Consider tuning alert to exclude modifications by SYSTEM account to known Microsoft task namespaces to reduce noise.

---

## 6. Tuning Recommendation

To reduce false positive volume from this detection without losing coverage:

```spl
index=winsec EventCode=4702
| where NOT (Account_Name="SYSTEM" AND TaskName LIKE "\Microsoft\Windows\%")
```

This excludes routine Windows maintenance while preserving alerts for any task modifications by non-SYSTEM accounts or outside the Microsoft namespace — where attacker activity is far more likely.

---

## 7. Lessons Learned

**The core false positive skill:**

Not every alert is a threat. A T1 analyst who escalates everything creates noise that desensitises Tier 2 and wastes response capacity. The discipline is knowing when the evidence *removes* suspicion, not just when it adds it.

For scheduled task alerts specifically, the four questions are always:
1. Who modified it? (SYSTEM vs. user account)
2. What process made the change? (svchost.exe vs. powershell.exe)
3. What is the task name? (Known Microsoft path vs. custom/random)
4. Is anything else suspicious in the same window?

If all four answers point to legitimate activity — close it and document why.
